6 - To do

This plugin is delivered as a proof of concept. Though it is a report framework on its own, it is squeezed into the user interface of the existing report framework. Some suggestions for further development.

a) Dedicated user interface

This plug-in would become easier to operate with a dedicated user interface.

The query options group should be replaced with edit boxes that allow to read and save plain text files. A single drop downlist can load a pair of built-in files in the edit-boxes. This list replaces the above overview of examples. The optional prefixes of the filenames indicate the required context (FAM/INDI) and rules (q/s for quick/slow). A file chooser per box can load and save custom queries and rules.

The output options group should use file choosers and these choosers should filter on the suggested extensions.

b) Get more from the semantic web

The semantic web uses triples of URI's: subject predicate object. An ontology (also expressed in triples) explains and defines types and meanings. An ontology also implies inference. The mother of all ontologies is OWL. At the moment applying full-fledged inference is slow. Common practice is to apply just the rules you actually need. This is a very brief introduction but it should have the most important pointers to start browsing the web about the subject.

The conversion does not comply to an established ontology. Cross site queries and inference would be easier when using something like FOAF or a similar ontology. The chosen format however was quick and easy to implement and is easy to recognise by GedCom tigers. Flexibility of the semantic web can deal which such an unorthodox approach. It might be wise to monitor for ontologies that might emerge and could replace the quick-and-dirty approach.

The power of the semantic web could become available when the plugin would allow to read more semantic data than just the converted GedCom. For example allow users to selectively download and collect whatever they want into a directory and read those files. A wizard could help with the selection.

Geonames is interesting for maps, see Amsterdam as an example

c) Disconnect conversion and reporting

If low priority background process are possible it might be an idea to perform conversion after saving the gedcom file. Performance of interactive functions should not suffer from such a background process. Escpecially because the slow rules may take minutes. Report generation should check if the last changes are saved and the conversion is completed.

It should also be possible to keep the rdf model synchronised. That would mean deleting all the tripples related to the changed entity or property and insert them all again.
