current status: proof of concept

This plugin is delivered as a proof of concept. Though it is a report framework on its own, part of the functionality is squeezed into the user interface of an existing report framework. The mashup functionality is even only available with bacth files.

design for the future

The functionality would be easier to use with a dedicated user interface. What is needed in my opinion:

  • options per gedcom

    • URI templates (family, individual, note, object, repository, source, sbumitter, place)
    • name and location of converted gedcom
    • name and location of the file with rules (allow to save/ choose the built-in examples)
    • name and location of collected mashup input data (preferably exported by the geo-plugin)
    • name and location of collected mashup output data
    • desired languages for dbpedia data (checkboxes for languages retuned by dbpediaLanguages.arq)
    • checkbox: save the converted gedcom whenever the gedcom itself is saved (especially with all the rules applied it takes a while)

  • menu comands

    • save converted gedcom
    • collect mashup data

  • report dialog (from the general menu or the context menu of an entity)

    • An entity chooser.
      When the dialog was opened from a context menu of an entity, that instance should be initialy presented. The chosen query may dictate another type of entity, The OK button should be disabled until tat conflict is resolved.
    • name and location of the query file (allow to save/choose the built-in examples)
    • checkbox 1: use converted gedcom
    • checkbox 2: use mashup output
    • name and location of the output file
    • name and location of the xsl file to insert into xml output (allow to save http://www.w3.org/TR/rdf-sparql-XMLres/result-to-html.xsl)
    One or both of the checkboxes 1 and 2 should be checked, selecting a built-in query automaticaly checks the apropriate ones. Prefixes of the built-in quiries indicate which rules should have been applied, disable the built in queries that need more rules than has been applied.

A mistake in a query might cause to execute forever. It should be executed in a subprocess that can be killed when the users chooses so. Same applies to conversion with custom rules.

Known bugs

  • You might have to run a context-free query before you can run a query that should be launched for an entity.
