6 - To do

This plugin is delivered as a proof of concept. Though it is a report framework on its own, it is squeezed into the user interface of the existing report framework. Some suggestions for further development.

a) Dedicated user interface

This plug-in would become easier to operate with a dedicated user interface.

The query options group should be replaced with edit boxes that allow to read and save plain text files. A single drop down list can load a pair of built-in files in the edit-boxes. This list replaces the above overview of examples. The optional prefixes of the filenames indicate the required context (FAM/INDI) and rules (q/s for quick/slow). A file chooser per box can load and save custom queries and rules.

The output options group should use file choosers and these choosers should filter on the suggested extensions.

b) Get more from the semantic web

A wizard could help to collect external data as described in the mashup section. The tools described there allow to disconnect conversion and reporting. Less stechnical users however may have problems using them.

c) Disconnect conversion and reporting

If low priority background process are possible it might be an idea to perform conversion after saving the GedCom file. Performance of interactive functions should not suffer from such a background process. Especially because the slow rules may take minutes. Report generation should check if the last changes are saved and the conversion is completed.

It should also be possible to keep the rdf model synchronised. That would mean deleting all the triples related to the changed entity or property and insert them all again.

d) Cancel query execution

A mistake in a query might cause to execute forever. It should be executed in a subprocess that can be killed when the users chooses so.

7 - Known bugs

  • You might have to run a context-free query before you can run a query that should be launched for an entity.
